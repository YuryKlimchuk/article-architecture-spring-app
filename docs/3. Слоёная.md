# 3. Слоёная архитектура (Layered)

## Как её только не называют

Прежде чем нырять — договоримся о терминах. У слоёной архитектуры много имён, и за каждым стоит примерно одно и то же:

- **Layered Architecture** — самый общий термин, от которого всё пошло.
- **N-tier Architecture** (N-звенная) — более формальное название. Когда говорят «N-tier», обычно имеют в виду физическое разделение слоёв по разным машинам.
- **Multi-tier / Multi-layered** — многоуровневая, многослойная.
- **Three-tier** (трёхзвенная) — частный случай N-tier с тремя слоями: презентация, логика, данные. Именно этот вариант мы и разбираем.
- В русском сообществе чаще всего говорят просто **«слоёная»**.

Исторически разница между *tier* и *layer* была важной: *tier* — это физически отдельная машина (клиент, сервер приложений, сервер БД), а *layer* — логический слой внутри одного приложения. Но со временем грань размылась, и сегодня оба слова используют как синонимы.

Теперь, когда с названиями ясно — откуда вообще взялась эта архитектура?

## Откуда ноги растут: книги и источники

Слоёная архитектура — ровесница enterprise-разработки. Если хочется первоисточников, вот три ключевых:

- **[Martin Fowler — Patterns of Enterprise Application Architecture](https://martinfowler.com/eaaCatalog/)** (2002). Фаулер не изобретал слоёную архитектуру, но каталогизировал паттерны, которые мы используем до сих пор: *Service Layer*, *Data Mapper*, *Repository*, *Active Record*. Именно из этой книги растут ноги у привычной связки Controller → Service → Repository.

- **[Eric Evans — Domain-Driven Design](https://www.amazon.com/Domain-Driven-Design-Tackling-Complexity-Software/dp/0321125215)** (2003). Эванс описал многоуровневую архитектуру в контексте DDD, где доменный слой изолирован от инфраструктуры. Это уже «слоёная с мозгами», но принцип вертикального разделения тот же.

- **[Microsoft Application Architecture Guide](https://learn.microsoft.com/en-us/previous-versions/msp-n-p/ff650706(v=pandp.10))** (2009). Классическое описание N-tier: Presentation → Business → Data. Не Java-источник, но повлиял на индустрию сильно — отсюда пошло само понятие «трёхзвенная архитектура», которое многие до сих пор отождествляют со слоёной.

Но давайте честно: большинство из нас не читало эти книги. Мы просто открыли Spring Boot, написали первый `@RestController`, и он заработал. Откуда же взялась слоёная архитектура на практике?

## Что имеем на практике

В типовом Spring-проекте слоёная архитектура выглядит так:

```
┌─────────────────────┐
│    Presentation     │  ← @RestController, DTO
├─────────────────────┤
│    Business /       │  ← @Service, бизнес-логика
│    Application      │
├─────────────────────┤
│    Data Access      │  ← @Repository, JPA Entity
└─────────────────────┘
```

Зависимости идут **сверху вниз**: контроллер знает о сервисе, сервис знает о репозитории, репозиторий знает о БД. Принцип простой: каждый слой может вызывать только слой непосредственно под ним.

На практике это означает структуру пакетов вида:

```
com.bookstore
├── controller
│   └── BookController
├── service
│   └── BookService
├── repository
│   └── BookRepository
└── model
    └── Book
```

А вот как это выглядит в коде — упрощённо, на примере нашего магазина книг:

```java
// Presentation Layer
@RestController
@RequestMapping("/books")
public class BookController {
    private final BookService bookService;

    @PostMapping
    public BookDto create(@RequestBody CreateBookRequest request) {
        return bookService.createBook(request);
    }
}

// Business / Application Layer
@Service
@Transactional
public class BookService {
    private final BookRepository bookRepository;

    public BookDto createBook(CreateBookRequest request) {
        Book book = new Book(request.title(), request.author(), request.price());
        book = bookRepository.save(book);
        return BookDto.from(book);
    }
}

// Data Access Layer
@Repository
public interface BookRepository extends JpaRepository<Book, Long> {
    List<Book> findByAuthor(String author);
}
```

Узнали свой код? Controller → Service → Repository — три слоя, зависимости строго вниз. На первых порах этого достаточно. Но смотрите, что здесь уже происходит:

- `BookService` **напрямую** вызывает `bookRepository.save(book)` — бизнес-логика жёстко прибита гвоздями к JPA.
- `Book` (доменный объект) одновременно является `@Entity` — JPA-аннотации живут прямо на классе, который должен представлять бизнес-правила.

Это и есть главная архитектурная уязвимость слоёной модели: **инфраструктура просачивается в домен**, и мы даже не замечаем.

Всё знакомо. Всё предсказуемо. И это не случайно.

## Почему слоёная — самая популярная

Слоёная архитектура не самая «правильная», но она — **самая популярная**. И вот почему:

### 1. Spring Boot толкает в неё по умолчанию

Откройте `start.spring.io`, добавьте Spring Web и Spring Data JPA — и у вас уже есть всё для слоёной архитектуры. Первый же гайд на [spring.io](https://spring.io/guides) показывает Controller → Service → Repository. Фреймворк не навязывает, но **подсказывает**.

### 2. Низкий порог входа

Слоёная архитектура интуитивно понятна. Она повторяет то, как мы привыкли думать о программах: «запрос пришёл → обработали → сохранили». Не нужно разбираться в доменных моделях, портах и инверсии зависимостей — достаточно знать, что `@Service` вызывает `@Repository`.

### 3. Естественная карта для веб-запроса

Жизненный цикл HTTP-запроса идеально ложится на три слоя: приняли запрос в контроллере → обработали бизнес-правила в сервисе → сходили в базу через репозиторий. Никакого лишнего умственного усилия.

### 4. Быстрый старт для MVP и прототипов

Когда нужно «показать работающий продукт через две недели», слоёная архитектура — лучший выбор. Она не требует проектирования интерфейсов, портов, адаптеров. Пишем код — он работает.

### 5. Все так делают

Это, пожалуй, главная причина. 90% статей, курсов и докладов по Spring показывают слоёную архитектуру. Junior приходит в проект и видит `controller`, `service`, `repository` — он уже знает, куда что класть. Команда не тратит время на обсуждение архитектурных правил, потому что «и так понятно».

## Для каких кейсов подходит

Слоёная архитектура — не «плохая». Она отлично работает, когда:

- **Простой CRUD.** Если бизнес-логика сводится к «создать, прочитать, обновить, удалить» и не содержит сложных правил — слоёной достаточно.

- **MVP и прототипы.** Когда важнее скорость выхода на рынок, а не идеальная архитектура.

- **Небольшая и средняя команда.** Когда все разработчики примерно одинакового уровня и понимают структуру «из коробки».

- **Стабильный стек.** Если вы уверены, что PostgreSQL (или что у вас там) никуда не денется в ближайшие пять лет, а внешние интеграции предсказуемы.

- **Микросервис на один домен.** Когда сервис отвечает за одну сущность и не содержит сложных межконтекстных взаимодействий.

## Где начинаются ограничения

Проблемы слоёной архитектуры не видны на старте. Они накапливаются по мере роста проекта. Вот первые тревожные звоночки:

### 1. Сервис знает о БД — и это проблема

В слоёной архитектуре `BookService` напрямую зависит от `BookRepository`, а тот — от JPA/Hibernate. Это значит, что бизнес-логика **привязана к способу хранения данных**. Поменять JPA на, скажем, MongoDB или REST-клиент — придётся переписывать сервис. Не потому что изменилась бизнес-логика, а потому что изменилась инфраструктура.

Но это ещё полбеды. Настоящая засада — в двойной роли доменного объекта:

```java
@Entity
@Table(name = "books")
public class Book {
    @Id @GeneratedValue
    private Long id;
    private String title;
    private String author;
    private BigDecimal price;
    // геттеры, сеттеры...
}
```

Класс `Book` одновременно и бизнес-сущность, и отражение таблицы в БД. `@Entity`, `@Table`, `@Id`, `@GeneratedValue` — это всё **инфраструктурные** аннотации, они описывают способ хранения, а не бизнес-правила. Но живут они прямо в домене.

Это как если бы паспорт гражданина содержал инструкцию для сканера в аэропорту: «меня считывать лазером с разрешением 300 dpi». К паспорту это отношения не имеет — а лежит внутри.

В слоёной архитектуре это считается нормой. В луковой и гексагональной — такое разделяют.

### 2. Сложно тестировать бизнес-логику изолированно

Хотите протестировать бизнес-правило «нельзя купить книгу, если её нет в наличии»? В слоёной архитектуре вам, скорее всего, придётся либо поднимать H2, либо мокать репозиторий. В обоих случаях тест `BookService` привязан к тому, **как** данные попадают в сервис, а не только к тому, **какие** они.

### 3. Рост сервиса в «божественный объект»

`BookService` начинается с метода `createBook()`. Потом добавляются `updateBook()`, `deleteBook()`, `findByAuthor()`. Потом появляется бизнес-правило «скидка для постоянных покупателей» — и оно тоже ложится в `BookService`. Потом нужно отправить email при создании книги — ещё один метод. Через полгода `BookService` — это 500 строк, и никто не помнит, за что он отвечает.

Слоёная архитектура не даёт механизма, который **принудительно** ограничивает размер сервисов. Всё держится на дисциплине команды. А дисциплина, как известно, заканчивается к третьему спринту с горящим дедлайном.

### 4. Сложно заменить внешнюю систему

Представьте, что вместо прямого вызова REST API платёжного шлюза нужно перейти на асинхронную очередь. В слоёной архитектуре вызов внешнего API часто зашит прямо в `@Service`. Чтобы заменить его, нужно лезть в бизнес-логику — а это всегда риск сломать правила, которые работали.

### 5. Смешение ответственности

Через год в `BookService` можно найти: бизнес-правила, валидацию, маппинг DTO, логику отправки уведомлений, вызов внешнего API и ещё что-то по мелочи. Класс делает всё — и за это отвечает. В маленьком проекте это не страшно. В большом — превращается в «страшно трогать, потому что всё развалится».

## Итог по слоёной

Слоёная архитектура — это **хороший старт для большинства проектов**. Она проста, понятна и подкреплена экосистемой Spring. Её проблема не в том, что она неправильная, а в том, что она **не защищает от собственного роста**.

Когда проект перерастает слоёную архитектуру, появляются два классических кандидата на замену: луковая и гексагональная. О них — следующие разделы.
