# 4. Луковая архитектура (Onion)

Термин **Onion Architecture** ввёл **Jeffrey Palermo** в 2008 году, устав от проектов, где смена ORM переписывала полкодовой базы.

## Ключевая идея: зависимости — внутрь

Если в слоёной архитектуре зависимости шли **сверху вниз** (контроллер → сервис → БД), то в луковой они направлены **снаружи внутрь**:

```
      ┌──────────────────────────────┐
      │         Infrastructure       │  ← JPA, REST-клиенты, Spring
      │   ┌──────────────────────┐   │
      │   │    Application       │   │  ← юзкейсы, порты
      │   │   ┌──────────────┐   │   │
      │   │   │   Domain     │   │   │  ← сущности, бизнес-правила
      │   │   └──────────────┘   │   │
      │   └──────────────────────┘   │
      └──────────────────────────────┘
```

Центр — **Domain**. Здесь живут бизнес-правила и сущности. Следующее кольцо — **Application**, оно содержит юзкейсы (сценарии использования). Самое внешнее кольцо — **Infrastructure** и **Presentation**: базы данных, REST-клиенты, контроллеры, Spring.

**Главное правило:** внутренние кольца ничего не знают о внешних. Domain не знает о JPA. Domain не знает о Spring. Domain — это чистый Java-код, который можно скомпилировать и протестировать без единой строчки конфигурации фреймворка.

## Как это работает: инверсия зависимостей

В слоёной архитектуре `BookService` напрямую зависит от `BookRepository`, который в свою очередь привязан к JPA. Стрелка: `Service → Repository → JPA`.

В луковой всё наоборот. Мы определяем **интерфейс репозитория в домене**, а реализацию — снаружи:

```
Domain: BookRepository (interface)  ←  Infrastructure: JpaBookRepository (реализация)
```

Стрелка развернулась: реализация зависит от интерфейса в домене, а не наоборот. Это и есть **Dependency Inversion Principle** — буква D из SOLID.

На уровне пакетов это выглядит так:

```
com.bookstore
├── domain
│   ├── model
│   │   └── Book               ← чистый POJO, никаких @Entity
│   └── port
│       └── BookRepository     ← интерфейс: что ДОЛЖЕН уметь репозиторий
├── application
│   └── BookService            ← юзкейс, зависит от BookRepository (интерфейса!)
├── infrastructure
│   └── JpaBookRepository      ← реализация BookRepository через JPA
└── presentation
    └── BookController         ← REST-контроллер
```

Ни один класс из `domain` не импортирует ничего из `infrastructure`. Ни один класс из `domain` не импортирует Spring. Домен — остров.

## Пример кода: тот же магазин книг, но в луковой

```java
// === DOMAIN: чистый POJO, никаких аннотаций ===
package com.bookstore.domain.model;

public class Book {
    private final Long id;
    private final String title;
    private final String author;
    private final BigDecimal price;

    // конструктор, геттеры, бизнес-методы...

    public boolean isExpensive() {
        return price.compareTo(new BigDecimal("1000")) > 0;
    }
}
```

```java
// === DOMAIN: интерфейс репозитория — здесь, в самом центре ===
package com.bookstore.domain.port;

public interface BookRepository {
    Book save(Book book);
    Optional<Book> findById(Long id);
    List<Book> findByAuthor(String author);
}
```

Обратите внимание: интерфейс `BookRepository` **не наследует** `JpaRepository`. Это не JPA-репозиторий. Это просто контракт: «я умею сохранять книги и находить их».

```java
// === APPLICATION: юзкейс, зависит только от интерфейса ===
package com.bookstore.application;

public class BookService {
    private final BookRepository bookRepository; // интерфейс, не реализация!

    public BookDto createBook(CreateBookRequest request) {
        Book book = new Book(null, request.title(), request.author(), request.price());
        book = bookRepository.save(book);
        return BookDto.from(book);
    }
}
```

`BookService` не знает, **как именно** `BookRepository` сохраняет книгу. JPA? MongoDB? Файл на диске? Не его дело. Он знает только контракт.

```java
// === INFRASTRUCTURE: реализация репозитория через JPA ===
package com.bookstore.infrastructure;

@Repository
public class JpaBookRepository implements BookRepository { // BookRepository из domain!
    private final JpaBookEntityRepository jpaRepo;
    private final BookMapper mapper;

    @Override
    public Book save(Book book) {
        BookEntity entity = mapper.toEntity(book);
        BookEntity saved = jpaRepo.save(entity);
        return mapper.toDomain(saved);
    }

    // ... остальные методы
}
```

А вот здесь уже живёт JPA. Отдельный класс `BookEntity` с `@Entity`, отдельный маппер туда-сюда. Домен чист, инфраструктура изолирована.

## Преимущества

### 1. Домен можно тестировать без Spring, без БД — и быстро

```java
class BookTest {
    @Test
    void expensiveBook() {
        Book book = new Book(1L, "Война и мир", "Толстой", new BigDecimal("1500"));
        assertTrue(book.isExpensive());
    }
}
```

Никаких `@SpringBootTest`, никакой H2, никаких моков. Обычный JUnit. Такой тест выполняется за миллисекунды.

### 2. Инфраструктура изолирована — замена и расширение без последствий

Решили переехать с JPA на MongoDB? Пишете `MongoBookRepository implements BookRepository` — и всё. Домен не трогаете. Нужно добавить кеширование? `CachedBookRepository implements BookRepository`. Домен не знает о кеше — и не должен.

### 3. Домен читается как бизнес-язык

Откройте пакет `domain.model` — там классы вроде `Book`, `Order`, `Customer`, методы вроде `isExpensive()`. Никаких `@Entity`, `@Table`, `@Column`. Бизнес-аналитик может прочитать и понять, что делает программа.

### 4. Тесты на бизнес-логику пишутся сразу

В слоёной архитектуре тест ждёт, пока готова вся цепочка: контроллер → сервис → репозиторий → БД. В луковой вы написали метод `calculateDiscount()` — и в ту же минуту покрыли тестом. Никакой БД, никакого Spring. К моменту готовности всей ручки бизнес-логика уже протестирована. Интеграционный тест — только на счастливый путь.

## Когда использовать

- **Сложные бизнес-правила.** Скидки, бонусы, правила отмены — слоёная тонет в if-else. Луковая заставляет вынести правила в домен.
- **Долгоживущий проект.** Код будет жить 2–3 года — смена БД, очереди, платёжного шлюза случится. Архитектура будет готова.
- **Несколько источников данных.** PostgreSQL + Elasticsearch + внешнее API — по одному адаптеру на каждый, домен в центре.

## Когда лучше НЕ использовать

- **Простой CRUD и MVP.** Бизнес-логика — одна строка `repository.save(...)`. Интерфейсы и адаптеры не окупятся. Слоёная быстрее.
- **Команда без опыта и жёсткие дедлайны.** Если половина не понимает, зачем интерфейс репозитория в `domain`, а дедлайн был вчера — будет хуже, чем аккуратная слоёная.

## Цена входа

- **Больше кода:** вместо трёх папок — пять, вместо одного класса — интерфейс + реализация + маппер.
- **Выше порог для новичков:** джуну придётся объяснять, почему интерфейс репозитория в `domain`.
- **Двойная модель:** доменная `Book` и `BookEntity` для JPA, плюс маппер между ними.

Но эта цена платится **один раз на старте**. Дальше архитектура работает на вас.
