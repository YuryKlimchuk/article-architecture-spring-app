# 4. Луковая архитектура (Onion)

## Как ещё называют и откуда взялась

Термин **Onion Architecture** ввёл **Jeffrey Palermo** в 2008 году — не книгой, а серией постов в личном блоге. Уже это намекает: луковая архитектура выросла не из академической теории, а из практической боли. Palermo устал от проектов, где смена ORM превращалась в переписывание половины кодовой базы, и предложил решение.

Позже идею подхватил **Robert C. Martin** в книге *Clean Architecture* (2017) и обобщил до более широкого принципа. Поэтому луковую иногда путают с Clean Architecture — они не одно и то же, но состоят в родстве. Clean Architecture — это зонтичная идея, а Onion — конкретная её реализация с акцентом на слои-кольца.

## Ключевая идея: зависимости — внутрь

Если в слоёной архитектуре зависимости шли **сверху вниз** (контроллер → сервис → БД), то в луковой они направлены **снаружи внутрь**:

```
      ┌──────────────────────────────┐
      │         Infrastructure       │  ← JPA, REST-клиенты, Spring
      │   ┌──────────────────────┐   │
      │   │    Application       │   │  ← юзкейсы, порты
      │   │   ┌──────────────┐   │   │
      │   │   │   Domain     │   │   │  ← сущности, бизнес-правила
      │   │   └──────────────┘   │   │
      │   └──────────────────────┘   │
      └──────────────────────────────┘
```

Центр — **Domain**. Здесь живут бизнес-правила и сущности. Следующее кольцо — **Application**, оно содержит юзкейсы (сценарии использования). Самое внешнее кольцо — **Infrastructure** и **Presentation**: базы данных, REST-клиенты, контроллеры, Spring.

**Главное правило:** внутренние кольца ничего не знают о внешних. Domain не знает о JPA. Domain не знает о Spring. Domain — это чистый Java-код, который можно скомпилировать и протестировать без единой строчки конфигурации фреймворка.

## Как это работает: инверсия зависимостей

В слоёной архитектуре `BookService` напрямую зависит от `BookRepository`, который в свою очередь привязан к JPA. Стрелка: `Service → Repository → JPA`.

В луковой всё наоборот. Мы определяем **интерфейс репозитория в домене**, а реализацию — снаружи:

```
Domain: BookRepository (interface)  ←  Infrastructure: JpaBookRepository (реализация)
```

Стрелка развернулась: реализация зависит от интерфейса в домене, а не наоборот. Это и есть **Dependency Inversion Principle** — буква D из SOLID.

На уровне пакетов это выглядит так:

```
com.bookstore
├── domain
│   ├── model
│   │   └── Book               ← чистый POJO, никаких @Entity
│   └── port
│       └── BookRepository     ← интерфейс: что ДОЛЖЕН уметь репозиторий
├── application
│   └── BookService            ← юзкейс, зависит от BookRepository (интерфейса!)
├── infrastructure
│   └── JpaBookRepository      ← реализация BookRepository через JPA
└── presentation
    └── BookController         ← REST-контроллер
```

Ни один класс из `domain` не импортирует ничего из `infrastructure`. Ни один класс из `domain` не импортирует Spring. Домен — остров.

## Пример кода: тот же магазин книг, но в луковой

```java
// === DOMAIN: чистый POJO, никаких аннотаций ===
package com.bookstore.domain.model;

public class Book {
    private final Long id;
    private final String title;
    private final String author;
    private final BigDecimal price;

    // конструктор, геттеры, бизнес-методы...

    public boolean isExpensive() {
        return price.compareTo(new BigDecimal("1000")) > 0;
    }
}
```

```java
// === DOMAIN: интерфейс репозитория — здесь, в самом центре ===
package com.bookstore.domain.port;

public interface BookRepository {
    Book save(Book book);
    Optional<Book> findById(Long id);
    List<Book> findByAuthor(String author);
}
```

Обратите внимание: интерфейс `BookRepository` **не наследует** `JpaRepository`. Это не JPA-репозиторий. Это просто контракт: «я умею сохранять книги и находить их».

```java
// === APPLICATION: юзкейс, зависит только от интерфейса ===
package com.bookstore.application;

public class BookService {
    private final BookRepository bookRepository; // интерфейс, не реализация!

    public BookDto createBook(CreateBookRequest request) {
        Book book = new Book(null, request.title(), request.author(), request.price());
        book = bookRepository.save(book);
        return BookDto.from(book);
    }
}
```

`BookService` не знает, **как именно** `BookRepository` сохраняет книгу. JPA? MongoDB? Файл на диске? Не его дело. Он знает только контракт.

```java
// === INFRASTRUCTURE: реализация репозитория через JPA ===
package com.bookstore.infrastructure;

@Repository
public class JpaBookRepository implements BookRepository { // BookRepository из domain!
    private final JpaBookEntityRepository jpaRepo;
    private final BookMapper mapper;

    @Override
    public Book save(Book book) {
        BookEntity entity = mapper.toEntity(book);
        BookEntity saved = jpaRepo.save(entity);
        return mapper.toDomain(saved);
    }

    // ... остальные методы
}
```

А вот здесь уже живёт JPA. Отдельный класс `BookEntity` с `@Entity`, отдельный маппер туда-сюда. Домен чист, инфраструктура изолирована.

## Преимущества

### 1. Домен можно тестировать без Spring, без БД

```java
class BookTest {
    @Test
    void expensiveBook() {
        Book book = new Book(1L, "Война и мир", "Толстой", new BigDecimal("1500"));
        assertTrue(book.isExpensive());
    }
}
```

Никаких `@SpringBootTest`, никакой H2, никаких моков. Обычный JUnit. Быстро. Надёжно.

### 2. Замена инфраструктуры не трогает бизнес-логику

Решили переехать с JPA на MongoDB? Пишете `MongoBookRepository implements BookRepository` — и всё. Домен не трогаете. `BookService` не меняется. Риск сломать бизнес-правила — ноль.

### 3. Домен читается как бизнес-язык

Откройте пакет `domain.model` — там классы с названиями вроде `Book`, `Order`, `Customer`, методы вроде `isExpensive()` или `isEligibleForDiscount()`. Никаких `@Entity`, `@Table`, `@Column`. Бизнес-аналитик может прочитать и понять, что делает программа. Java-разработчик, который только пришёл в проект, тоже.

### 4. Точки расширения определены явно

Интерфейс `BookRepository` — это единственное место, через которое домен общается с внешним миром. Если нужно добавить кеширование — пишете `CachedBookRepository implements BookRepository` и оборачиваете вызовы. Домен не знает о кеше. Не знает и не должен.

## Когда использовать обязательно

- **Сложная бизнес-логика.** Если в проекте есть правила вроде «скидка 10% для постоянных клиентов, но не суммируется с акционной ценой, и только если книга не из списка исключений» — это звоночек. Слоёная утонет в if-else внутри сервиса. Луковая заставит вынести правила в домен.

- **Долгоживущий проект.** Если вы рассчитываете, что код будет жить 2–3 года и больше — луковая окупается. Поменять БД, добавить очередь, сменить платёжный шлюз — всё это случится, и архитектура будет готова.

- **Несколько источников данных.** Книги хранятся в PostgreSQL, отзывы — в Elasticsearch, цены подтягиваются из внешнего API. В слоёной это ад. В луковой — по одному адаптеру на каждый источник, домен в центре, всё чисто.

- **Микросервисы со сложной доменной логикой.** Если микросервис — это не просто прокси к БД, а настоящий домен с правилами.

## Когда лучше НЕ использовать

- **Простой CRUD.** Если весь сервис — это создать/прочитать/обновить/удалить, луковая архитектура будет overengineering-ом. Вы напишете интерфейсы, адаптеры, мапперы — а бизнес-логика так и останется в одну строку `repository.save(...)`. Овчинка не стоит выделки.

- **Прототип или MVP на 2–3 недели.** Когда нужно показать работающий продукт и получить фидбек, архитектурная чистота не главное. Слоёная быстрее в разработке.

- **Команда без опыта в DDD/портах/адаптерах.** Луковая требует дисциплины. Если половина команды не понимает, зачем интерфейс репозитория лежит в `domain`, а реализация — в `infrastructure`, вы потратите время на споры, а не на код. Результат будет хуже, чем просто слоёная, сделанная аккуратно.

- **Один разработчик и жёсткие дедлайны.** Если вы один тащите проект и дедлайн был вчера — пишите слоёную. Луковая требует чуть больше времени на старте, и это время должно быть.

## Цена входа

Давайте честно: у луковой архитектуры есть цена. Вот что придётся заплатить:

- **Больше пакетов, больше интерфейсов, больше церемоний.** Вместо трёх папок (`controller/service/repository`) — четыре или пять, с вложенными подпакетами. Вместо одного класса для работы с БД — интерфейс в домене + реализация в инфраструктуре + маппер.

- **Выше порог входа для новичков.** Джун, который привык к «контроллер вызывает сервис, сервис вызывает репозиторий», споткнётся о «почему интерфейс репозитория в домене?». Надо будет объяснять.

- **Маппинг туда-сюда.** В слоёной `Book` = JPA-сущность, одна модель на всё. В луковой как минимум две: доменная `Book` и `BookEntity` для JPA. Плюс маппер между ними. Это дополнительные классы и дополнительная работа.

Но вот что важно: эта цена платится **один раз на старте**. Дальше архитектура работает на вас — а не вы на неё.

## Отличие от слоёной в одной фразе

Слоёная: **зависимости сверху вниз** — контроллер знает о сервисе, сервис знает о БД.

Луковая: **зависимости снаружи внутрь** — всё зависит от домена, домен не зависит ни от чего.
