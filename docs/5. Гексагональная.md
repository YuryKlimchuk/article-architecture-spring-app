# 5. Гексагональная архитектура (Hexagonal / Ports & Adapters)

Термин ввёл **Alistair Cockburn** в 2005 году. Шестиугольник — не про число портов, а про то, что «портов может быть сколько угодно». Второе название — **Ports & Adapters** — пожалуй, точнее.

## Ключевая идея: порты и адаптеры

Приложение общается с внешним миром только через **порты** (ports). К каждому порту снаружи подключается **адаптер** (adapter) — конкретная реализация технологии.

```
Внешний мир                           Приложение                         Внешний мир
  (driving)                              (ядро)                            (driven)

┌───────────┐                                                   ┌───────────┐
│ REST      │──┐                                            ┌──│ PostgreSQL│
│ Controller│  │    ┌──────────────────────────────┐       │  │ (JPA)    │
└───────────┘  │    │                              │       │  └───────────┘
               ├────┤   входящий порт    исх. порт ├───────┤
┌───────────┐  │    │   (CreateBook)     (BookRepo)│       │  ┌───────────┐
│ gRPC      │──┤    │                              │       ├──│ REST      │
│ Handler   │  │    │          ДОМЕН               │       │  │ Client    │
└───────────┘  │    │                              │       │  └───────────┘
               ├────┤                              ├───────┤
┌───────────┐  │    │                              │       │  ┌───────────┐
│ CLI       │──┘    └──────────────────────────────┘       └──│ Message   │
│ Command   │                                                  │ Queue     │
└───────────┘                                                  └───────────┘

Driving side (левая сторона):              Driven side (правая сторона):
мир дёргает приложение                     приложение дёргает мир
```

**Driving side** (левая, входящая) — это как внешний мир инициирует общение с приложением: REST-контроллер, gRPC-обработчик, CLI-команда, UI. Каждый вход подключается к одному и тому же входящему порту.

**Driven side** (правая, исходящая) — это как приложение само инициирует общение с внешним миром: БД, внешний API, очередь сообщений, файловая система. К каждому исходящему порту можно подключить свой адаптер.

Эта симметрия — ключевая особенность гексагональной архитектуры. У луковой такого явного деления на левую и правую сторону нет.

## Структура пакетов

```java
com.bookstore
├── domain                        // чистое ядро
│   ├── model
│   │   └── Book                  // POJO, никаких аннотаций
│   └── service
│       └── BookService           // доменный сервис (опционально)
├── application                   // порты и юзкейсы
│   ├── port
│   │   ├── in
│   │   │   └── CreateBookUseCase // входящий порт: что умеет приложение
│   │   └── out
│   │       └── BookRepository    // исходящий порт: что нужно приложению
│   └── service
│       └── CreateBookService     // реализация входящего порта (юзкейс)
├── adapter
│   ├── in
│   │   └── BookRestController    // REST-адаптер для входящего порта
│   └── out
│       ├── JpaBookRepository     // JPA-адаптер для исходящего порта
│       ├── BookEntity            // JPA-сущность (только здесь!)
│       └── BookMapper            // маппер entity ↔ domain
```

Заметьте: `domain` и `application/port` не зависят ни от Spring, ни от JPA. `adapter` — зависит от всего и подключает приложение к реальному миру.

## Пример кода: магазин книг в гексагональной

### Port IN — что умеет приложение

```java
package com.bookstore.application.port.in;

public interface CreateBookUseCase {
    BookDto create(CreateBookCommand command);
}
```

Входящий порт: «чтобы создать книгу, дёрни этот метод». REST, gRPC или CLI — безразлично.

### Port OUT — что нужно приложению

```java
package com.bookstore.application.port.out;

public interface BookRepository {
    Book save(Book book);
    Optional<Book> findById(Long id);
}
```

Исходящий порт: «мне нужно сохранять и находить книги». JPA, MongoDB, in-memory — безразлично.

### Реализация юзкейса

```java
package com.bookstore.application.service;

public class CreateBookService implements CreateBookUseCase {
    private final BookRepository bookRepository; // port OUT, интерфейс!

    @Override
    public BookDto create(CreateBookCommand command) {
        Book book = new Book(null, command.title(), command.author(), command.price());
        book = bookRepository.save(book);
        return BookDto.from(book);
    }
}
```

Знает только порты. Никаких деталей реализации.

### Adapter IN — REST

```java
package com.bookstore.adapter.in;

@RestController
@RequestMapping("/books")
public class BookRestController {
    private final CreateBookUseCase createBookUseCase; // port IN

    @PostMapping
    public BookDto create(@RequestBody CreateBookCommand command) {
        return createBookUseCase.create(command);
    }
}
```

Тонкий контроллер: принял HTTP-запрос → передал в порт. Бизнес-логики — ноль.

### Adapter OUT — JPA

```java
package com.bookstore.adapter.out;

@Repository
public class JpaBookRepository implements BookRepository {
    // ... JPA-реализация — точно такая же, как в луковой
    // BookEntity, мапперы — всё здесь, в адаптере
}
```

JPA заперт в адаптере. Домен не видит.

## Чем гексагональная отличается от луковой

Гексагональная и луковая близки — домен в центре, изоляция инфраструктуры, инверсия зависимостей. Но разница есть:

| Луковая (Onion) | Гексагональная (Hexagonal) |
|---|---|
| Акцент на **слоях-кольцах** и их изоляции | Акцент на **портах и адаптерах** |
| Нет явного разделения на входящее/исходящее | Явное разделение: **driving** vs **driven** |
| Говорит: «Domain в центре, вокруг — приложение, снаружи — инфраструктура» | Говорит: «Приложение общается через порты, подключайте любые адаптеры» |
| Метафора: луковица, кольца | Метафора: шестиугольник с портами на гранях |
| Хороша для проектов с чёткой иерархией зависимостей | Хороша для систем с **множеством интеграций** |

Коротко: **луковая про слои, гексагональная про коннекторы**. Они не противоречат друг другу. Гексагональная лучше там, где входов и выходов много: REST + gRPC + Kafka + БД + внешний API. Каждый новый канал — ещё один адаптер, ядро не меняется.

## Преимущества

### 1. Тестируемость, отложенные решения, замена без боли

Протестировать приложение? Подключите in-memory заглушку вместо реального адаптера — ни Spring, ни БД. Не знаете, какая будет БД на старте? Напишите `InMemoryBookRepository`, реализуйте домен, а через месяц добавьте JPA-адаптер. Переехали с REST на gRPC? Новый `BookGrpcEndpoint` — юзкейс не меняется.

### 2. Явные границы для команды

Структура пакетов `port/in`, `port/out`, `adapter/in`, `adapter/out` сама диктует, куда что класть. Контракт между разработчиками живёт в коде, а не в вики.

## Когда использовать

- **Множество интеграций.** REST + gRPC + Kafka + несколько БД + внешние API. Каждый новый канал — адаптер, ядро стабильно.
- **Несколько UI к одному домену.** REST для мобильного приложения и gRPC для внутренних сервисов — два адаптера к одному порту.
- **DDD и событийные микросервисы.** Гексагональная и DDD — классическая пара. Входящий порт — подписка на очередь, исходящий — публикация событий.

## Когда НЕ использовать

- **Простой CRUD и MVP.** Один контроллер, одна БД — хватит слоёной. Скорость важнее архитектурной чистоты.
- **Один канал входа/выхода и команда без опыта.** Только REST + PostgreSQL — гексагональная не раскроется, луковая проще. Терминология «порт/адаптер/driving/driven» — лишний барьер для новичков.

## Цена входа

- **Больше кода:** каждый порт — интерфейс, каждый адаптер — реализация, плюс мапперы.
- **Новая терминология:** driving/driven, port, adapter — придётся объяснять коллегам.
- **Выше порог для новичков:** структура пакетов не «как в туториале по Spring».
