# 5. Гексагональная архитектура (Hexagonal / Ports & Adapters)

## Откуда взялась и почему «гексагональная»

В 2005 году **Alistair Cockburn** — один из авторов Agile-манифеста, практик, не академик — написал статью на своём сайте с заголовком «Hexagonal Architecture». Идея была простая: приложение — это шестиугольник в центре, а грани — это порты, через которые оно общается с миром.

Почему шестиугольник? Не потому что портов ровно шесть — Cockburn специально это подчеркнул:

> «Шестиугольник позволяет нарисовать столько портов с каждой стороны, сколько нужно, и оставляет место между ними. Четырёхугольник намекает на четыре стороны, трёхугольник — на три. Шестиугольник ни на что не намекает».

Гениально же. Форма фигуры — это способ сказать: «портов может быть сколько угодно, не зацикливайтесь на числе».

С тех пор у архитектуры два равноправных названия: **Hexagonal** (форма) и **Ports & Adapters** (суть). Второе, пожалуй, точнее.

## Ключевая идея: порты и адаптеры

Приложение общается с внешним миром только через **порты** (ports). К каждому порту снаружи подключается **адаптер** (adapter) — конкретная реализация технологии.

```
Внешний мир                           Приложение                         Внешний мир
  (driving)                              (ядро)                            (driven)

┌───────────┐                                                   ┌───────────┐
│ REST      │──┐                                            ┌──│ PostgreSQL│
│ Controller│  │    ┌──────────────────────────────┐       │  │ (JPA)    │
└───────────┘  │    │                              │       │  └───────────┘
               ├────┤   входящий порт    исх. порт ├───────┤
┌───────────┐  │    │   (CreateBook)     (BookRepo)│       │  ┌───────────┐
│ gRPC      │──┤    │                              │       ├──│ REST      │
│ Handler   │  │    │          ДОМЕН               │       │  │ Client    │
└───────────┘  │    │                              │       │  └───────────┘
               ├────┤                              ├───────┤
┌───────────┐  │    │                              │       │  ┌───────────┐
│ CLI       │──┘    └──────────────────────────────┘       └──│ Message   │
│ Command   │                                                  │ Queue     │
└───────────┘                                                  └───────────┘

Driving side (левая сторона):              Driven side (правая сторона):
мир дёргает приложение                     приложение дёргает мир
```

**Driving side** (левая, входящая) — это как внешний мир инициирует общение с приложением: REST-контроллер, gRPC-обработчик, CLI-команда, UI. Каждый вход подключается к одному и тому же входящему порту.

**Driven side** (правая, исходящая) — это как приложение само инициирует общение с внешним миром: БД, внешний API, очередь сообщений, файловая система. К каждому исходящему порту можно подключить свой адаптер.

Эта симметрия — ключевая особенность гексагональной архитектуры. У луковой такого явного деления на левую и правую сторону нет.

## Структура пакетов

```java
com.bookstore
├── domain                        // чистое ядро
│   ├── model
│   │   └── Book                  // POJO, никаких аннотаций
│   └── service
│       └── BookService           // доменный сервис (опционально)
├── application                   // порты и юзкейсы
│   ├── port
│   │   ├── in
│   │   │   └── CreateBookUseCase // входящий порт: что умеет приложение
│   │   └── out
│   │       └── BookRepository    // исходящий порт: что нужно приложению
│   └── service
│       └── CreateBookService     // реализация входящего порта (юзкейс)
├── adapter
│   ├── in
│   │   └── BookRestController    // REST-адаптер для входящего порта
│   └── out
│       ├── JpaBookRepository     // JPA-адаптер для исходящего порта
│       ├── BookEntity            // JPA-сущность (только здесь!)
│       └── BookMapper            // маппер entity ↔ domain
```

Заметьте: `domain` и `application/port` не зависят ни от Spring, ни от JPA. `adapter` — зависит от всего и подключает приложение к реальному миру.

## Пример кода: магазин книг в гексагональной

### Port IN — что умеет приложение

```java
package com.bookstore.application.port.in;

public interface CreateBookUseCase {
    BookDto create(CreateBookCommand command);
}
```

Это входящий порт. Он говорит миру: «чтобы создать книгу, дёрни этот метод». Как именно мир доставит запрос — REST, gRPC, CLI — порту безразлично.

### Port OUT — что нужно приложению

```java
package com.bookstore.application.port.out;

public interface BookRepository {
    Book save(Book book);
    Optional<Book> findById(Long id);
}
```

Исходящий порт. Он говорит: «мне нужно уметь сохранять и находить книги». Как именно — JPA, MongoDB, заглушка в памяти — безразлично.

### Реализация юзкейса

```java
package com.bookstore.application.service;

public class CreateBookService implements CreateBookUseCase {
    private final BookRepository bookRepository; // port OUT, интерфейс!

    @Override
    public BookDto create(CreateBookCommand command) {
        Book book = new Book(null, command.title(), command.author(), command.price());
        book = bookRepository.save(book);
        return BookDto.from(book);
    }
}
```

`CreateBookService` знает только порты — входящий реализует, исходящий дёргает. Никаких деталей.

### Adapter IN — REST

```java
package com.bookstore.adapter.in;

@RestController
@RequestMapping("/books")
public class BookRestController {
    private final CreateBookUseCase createBookUseCase; // port IN

    @PostMapping
    public BookDto create(@RequestBody CreateBookCommand command) {
        return createBookUseCase.create(command);
    }
}
```

REST-контроллер живёт в адаптерах. Он тонкий — только принимает HTTP-запрос и передаёт в порт. Бизнес-логики — ноль.

### Adapter OUT — JPA

```java
package com.bookstore.adapter.out;

@Repository
public class JpaBookRepository implements BookRepository {
    // ... JPA-реализация — точно такая же, как в луковой
    // BookEntity, мапперы — всё здесь, в адаптере
}
```

JPA заперт в адаптере. Домен его не видит.

## Чем гексагональная отличается от луковой

Вот здесь многие спотыкаются — и это нормально. Гексагональная и луковая действительно очень близки. Обе ставят домен в центр. Обе изолируют инфраструктуру. Обе используют инверсию зависимостей. Но разница есть:

| Луковая (Onion) | Гексагональная (Hexagonal) |
|---|---|
| Акцент на **слоях-кольцах** и их изоляции | Акцент на **портах и адаптерах** |
| Нет явного разделения на входящее/исходящее | Явное разделение: **driving** vs **driven** |
| Говорит: «Domain в центре, вокруг — приложение, снаружи — инфраструктура» | Говорит: «Приложение общается через порты, подключайте любые адаптеры» |
| Метафора: луковица, кольца | Метафора: шестиугольник с портами на гранях |
| Хороша для проектов с чёткой иерархией зависимостей | Хороша для систем с **множеством интеграций** |

Если совсем грубо: **луковая про слои, гексагональная про коннекторы**. Они не противоречат друг другу — многие проекты используют идеи из обеих одновременно.

Но гексагональная лучше справляется там, где входов и выходов много: REST + gRPC + Kafka + БД + внешний API + файловая система. Каждый новый канал — это просто ещё один адаптер. Ядро не меняется.

## Преимущества

### 1. Тестируемость на максималках

Можно протестировать приложение, подключив к портам заглушки вместо реальных адаптеров:

```java
class CreateBookServiceTest {
    private final BookRepository stubRepo = new InMemoryBookRepository();
    private final CreateBookUseCase useCase = new CreateBookService(stubRepo);

    @Test
    void shouldCreateBook() {
        BookDto result = useCase.create(new CreateBookCommand("Война и мир", ...));
        assertEquals("Война и мир", result.title());
    }
}
```

Ни Spring, ни БД. In-memory адаптер, который вы написали за 5 минут.

### 2. Отложенные решения

На старте проекта вы можете не знать, какая будет БД. Напишите `InMemoryBookRepository`, реализуйте весь домен и юзкейсы, протестируйте — а через месяц добавьте JPA-адаптер. Ядро не трогаете.

### 3. Замена адаптера без последствий

Переехали с REST на gRPC? Новый `BookGrpcEndpoint` адаптер — и всё. Старый REST-адаптер можно выключить. Юзкейс `CreateBookUseCase` не меняется.

### 4. Явные границы для команды

Структура пакетов `port/in`, `port/out`, `adapter/in`, `adapter/out` сама диктует, куда что класть. Не надо договариваться — архитектура подсказывает. Контракт между разработчиками живет в коде, а не в вики.

## Когда использовать

- **Множество интеграций.** REST + gRPC + Kafka + несколько БД + внешние API. Каждый новый канал — адаптер. Ядро стабильно.
- **Несколько UI-интерфейсов к одному домену.** REST API для мобильного приложения и gRPC для внутренних сервисов — два адаптера к одному порту.
- **Частая смена протоколов.** Сегодня REST, завтра gRPC, послезавтра GraphQL. Меняете адаптер — домен не трогаете.
- **Микросервисы с событиями.** Входящий порт — подписка на очередь, исходящий — публикация событий. Гексагональная ложится идеально.
- **DDD-проекты.** Гексагональная и DDD — классическая пара. DDD отвечает за то, как устроен домен, гексагональная — как домен подключён к миру.

## Когда НЕ использовать

- **Простой CRUD.** Один контроллер, одна БД — хватит слоёной.
- **Прототипы и MVP.** Скорость разработки важнее архитектурной чистоты.
- **Один канал входа и один канал выхода.** Если у вас только REST и только PostgreSQL — гексагональная не раскроет своих преимуществ. Луковая уже даст достаточно изоляции, и проще.
- **Команда без опыта.** Терминология «порт/адаптер/driving/driven» — дополнительный барьер. Если команда не понимает зачем, архитектура превращается в формальность.

## Цена входа

- **Порты + адаптеры + маппинг** — больше кода, чем в слоёной. Каждый порт — это интерфейс. Каждый адаптер — реализация. Плюс мапперы.
- **Новая терминология.** Driving/driven side, port, adapter — коллеги могут спросить «а зачем эти слова?». Придётся объяснять, и это нормально.
- **Выше порог для новичков.** Структура пакетов не «как в туториале по Spring». Нужно время, чтобы освоиться.
