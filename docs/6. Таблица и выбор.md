# 6. Сравнительная таблица и что выбрать

Мы разобрали три архитектуры на одном примере. Теперь собираем всё в таблицу и переходим к главному: что выбрать в своём проекте.

## Таблица
| Критерий | Слоёная (Layered) | Луковая (Onion) | Гексагональная (Hexagonal) |
|---|---|---|---|
| **Направление зависимостей** | Сверху вниз: контроллер → сервис → БД | Снаружи внутрь: всё зависит от домена | Всё зависит от портов, адаптеры снаружи |
| **Центр архитектуры** | Слои равноправны | Домен — центр | Приложение с портами — центр |
| **Домен знает о БД?** | Да, через `@Entity` и `JpaRepository` | Нет, интерфейс репозитория в домене | Нет, исходящий порт — только интерфейс |
| **Замена БД** | Требует правки сервиса и домена | Только новый адаптер, домен не трогаем | Только новый адаптер для исходящего порта |
| **Тестирование домена** | Нужна H2 или моки БД | Чистый JUnit, без Spring и БД | Чистый JUnit, c in-memory адаптером |
| **Количество слоёв / элементов** | 3 слоя | 4 кольца (Domain → Application → Infra → Presentation) | Порты (in/out) + адаптеры + домен |
| **Структура пакетов** | `controller` / `service` / `repository` | `domain` / `application` / `infrastructure` / `presentation` | `domain` / `application/port/in` / `application/port/out` / `adapter/in` / `adapter/out` |
| **Порог входа** | Низкий — всё как в Spring-туториалах | Средний — нужно понять инверсию зависимостей | Выше среднего — новая терминология |
| **Когда использовать** | CRUD, MVP, прототипы, малая команда | Сложный домен, долгий проект, несколько БД | Множество интеграций, несколько UI, DDD-проекты |
| **Ключевая метафора** | Слоёный пирог | Луковица | Шестиугольник с портами |

Если запомнить из всей статьи только одно — запомните это:

> В слоёной архитектуре зависимости идут **сверху вниз** — от контроллера к БД. Домен знает о БД.
>
> В луковой и гексагональной зависимости идут **снаружи внутрь** — к домену. Домен не знает ни о чём.

Всё остальное — детали. Кольца, порты, адаптеры, аннотации — инструменты для этой инверсии.

## Какую выбрать
Ни одна из трёх архитектур не «лучшая вообще». Выбор — trade-off между скоростью сейчас и гибкостью потом.

```
Ваш проект — это...

CRUD без сложной логики? ──→ Слоёная (Layered)
  Пример: каталог книг, админка, REST-прокси к БД

Сложные бизнес-правила? ──→ Луковая (Onion)
  Пример: маркетплейс, страховой движок, биллинг

Много интеграций, несколько UI? ──→ Гексагональная (Hexagonal)
  Пример: микросервис с REST + gRPC + Kafka + 3 БД

И сложный домен, И много интеграций? ──→ Луковая + элементы гексагональной
  Да, их можно и нужно комбинировать
```

### Слоёная
Идеальный кандидат — сервис каталога книг. Запрос → нашли в БД → вернули JSON. Бизнес-правила заканчиваются на «цена не может быть отрицательной».

**Плюсы:** быстрая разработка, низкий порог входа, минимум церемоний.
**Минусы:** проект растёт → сервисы-монстры, замена БД ломает бизнес-логику, тесты медленные.

### Луковая
Идеальный кандидат — сервис заказов. Скидки, бонусы, правила отмены — бизнес-правила сложные и меняются часто. Луковая даёт **домен, независимый от инфраструктуры**.

**Плюсы:** домен тестируется быстро без БД, замена БД = новый адаптер, бизнес-логика читается как бизнес-логика.
**Минусы:** больше кода на старте, выше порог входа, для простого CRUD — overengineering.

#### «Мы никогда не будем менять базу» — и что?

Часто слышу: *«Зачем нам луковая архитектура, мы не собираемся менять БД»*. Это возражение бьёт мимо цели.

Главная ценность луковой (и гексагональной) архитектуры — **не в возможности однажды выдернуть JPA и воткнуть MongoDB**. Сама по себе замена БД случается редко. Но архитектура, которая к этому готова, даёт гораздо больше:

- **Бизнес-логика тестируется мгновенно**, без поднятия H2 или Testcontainers. Каждый тест — доли секунды вместо секунд.
- **Домен читается как обычный текст**, а не как простыня `@Entity`, `@Table`, `@Column`. Бизнес-аналитик или новый разработчик понимает, что делает код, не зная JPA.
- **Структура пакетов сама диктует, куда класть новый код.** Не надо держать в голове договорённости — `domain` для правил, `infrastructure` для технических деталей.
- **Любое изменение изолировано.** Не только смена БД, но и смена API, добавление очереди, переход на другую библиотеку — бизнес-правила не трогаются.

Иными словами: архитектура, готовая к замене БД — это **побочный эффект хорошей структуры**, а не самоцель. Вы платите один раз на старте и получаете код, который легко менять. Что именно менять — БД, фреймворк, протокол — не так важно. Важно, что менять не больно.

### Гексагональная
Идеальный кандидат — сервис уведомлений. REST + очередь + внешний API + БД. Много входов, много выходов. Каждый новый канал — адаптер, ядро не меняется.

**Плюсы:** подключение канала = новый адаптер, можно начать с in-memory заглушек, явные границы в структуре пакетов.
**Минусы:** тяжеловесная терминология, если вход и выход по одному — не раскроется.

### Луковая или гексагональная?
Они близки. Правило простое:

- Думаете «изолировать домен от инфраструктуры» → **луковая**.
- Думаете «подключать и отключать разные каналы входа и выхода» → **гексагональная**.

На практике большинство проектов берут среднее: домен в центре (луковая идея), интеграции через адаптеры к портам (гексагональная идея). Не бойтесь смешивать.

### Можно ли начать со слоёной и мигрировать потом?
Технически — да. Практически — **больно**.

Переход со слоёной на луковую: вытащить JPA из домена, разделить `Book` на `Book` и `BookEntity`, перевернуть зависимости, переписать тесты. Это архитектурная миграция, а не рефакторинг за спринт.

Честная рекомендация: **если уже сейчас видите, что проект будет жить долго и обрастать сложностью — начните с луковой (или гексагональной).** Стартовые затраты окупятся через 3–6 месяцев.

Если прототип, и вы честно не знаете, взлетит ли — слоёной достаточно. Оставьте в коде `// TODO: migrate to Onion if this thing survives` — через год скажут спасибо.
