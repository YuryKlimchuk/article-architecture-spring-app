###Название статьи 
**«Layered, Onion, Hexagonal архитектуры: в чём разница? Разбираем на примерах Spring»»

###Предлагаемый план статьи
1. Введение — проблема: три названия, одна путаница. Спойлер: разница в направлении зависимостей.

2. Почему их путают? — общее для всех трёх: разделение на слои, изоляция бизнес-логики, независимость от фреймворков в теории. Именно из-за этих общих черт новички думают, что это одно и то же.

3. Слоёная (Layered) — классика Spring: Controller → Service → Repository. Код примера (создание книги). Плюсы и главный минус: сервис знает о БД.

4. Луковая (Onion) — домен в центре, зависимости внутрь. Тот же пример, но интерфейс репозитория в ядре, реализация снаружи. В чём отличие от слоёной.

4. Гексагональная (Hexagonal / Ports & Adapters) — входящие и исходящие порты. Тот же пример, но с явным разделением. В чём отличие от луковой (многие их путают).

5. Сравнительная таблица — кратко, по 5–6 ключевым критериям.

6. Какую выбрать для Spring-проекта? — практические рекомендации: для простого CRUD, для сложного домена, для интеграций.

7. Заключение — главная мысль: дело не в количестве слоёв, а в том, куда смотрят зависимости.
